## Слайд 3. Архитектура решения
Клиент (браузер) отправляет HTTP‑запрос на Ingress (nginx), размещённый в публичной подсети нашей VPC. Ingress‑контроллер разбирает путь:

- `/*` направляется на frontend‑ingress → 2 реплики статического фронтенда;
- `/api/*` направляется на api‑ingress, откуда запросы уходят на `user‑service` (2 реплики) или `ticket‑service` (2 реплики) в зависимости от эндпоинта.

`ticket‑service` при необходимости сам обращается к `user‑service` — например, чтобы проверить, привязан ли билет к конкретному пользователю, перед тем как подтвердить операцию.

В качестве общего хранилища используется `Redis` (1 реплика) — туда оба сервиса пишут и читают данные (сессии, кэш билетов и т.п.).

В кластере настроены два объекта `HorizontalPodAutoscaler` (HPA) — для user-service и ticket-service. Они автоматически увеличивают или уменьшают количество реплик в зависимости от загрузки CPU (задан порог, например 50%). Это позволяет сервисам выдерживать всплески трафика без ручного вмешательства.

Минимум две реплики гарантируют отказоустойчивость (при падении одной пода вторая продолжает обслуживать запросы) и позволяют равномерно распределять нагрузку. Redis развёрнут в единственном экземпляре, поскольку для демонстрации функциональности этого достаточно, а кластеризация Redis выходит за рамки базового сценария.

## Слайд 4. Реализация: микросервисы, Docker Hub, Terraform, Kubernetes

В рамках проекта реализовано три микросервиса:

- **User Service** — обрабатывает регистрацию и аутентификацию пользователей. Поддерживает методы `POST /api/users/register`, `POST /api/users/login`, а также `GET /api/users/health` для проверки работоспособности.
- **Ticket Service** — управляет билетами. Предоставляет `POST /api/tickets/create`, `GET /api/tickets/list`, `DELETE /api/tickets/{id}`. При необходимости сам обращается к User Service для проверки привязки билета к пользователю.
- **Frontend** — статическое приложение (HTML/CSS/JS), которое через браузер взаимодействует с API.

Готовые Docker‑образы этих сервисов собраны и загружены на Docker Hub под именами `fernandoprod/user-service`, `fernandoprod/ticket-service` и `fernandoprod/frontend`. Это позволяет разворачивать их в любой среде простым указанием тега образа.

Инфраструктура описана в виде Terraform‑файлов. В них мы зафиксировали создание VPC, публичной и приватной подсетей, роутера, сетевых интерфейсов и группы безопасности `microservices-k8s-sg`. Такой подход гарантирует полную воспроизводимость окружения по команде `terraform apply`.

Для оркестрации написаны Kubernetes‑манифесты (YAML), которые определяют:

- Namespace `microservices`;
- Deployment и Service для каждого из трёх приложений;
- Deployment и Service для Redis;
- Ingress‑ресурсы, разделяющие трафик на фронтенд и API;
- HorizontalPodAutoscaler для `user-service` и `ticket-service`.

Проверка корректности развёртывания выполняется командой:

    ```bash
    kubectl get deployments -n microservices

Она показывает, что все деплойменты находятся в состоянии READY (например, 2/2 реплики) и подтверждает, что система запущена и готова к работе.

## Слайд 6. Проверка автомасштабирования (HPA)
Мы открыли два терминала:
1. В первом запустили нагрузочный тест с помощью утилиты hey:
    ```bash
    hey -z 60s -c 20 http://89.208.208.36.nip.io/api/users/health
2. Во втором параллельно отслеживали поды:
    ```bash
    kubectl get pods -n microservices -w```
   
Под нагрузкой HPA для user-service зафиксировал рост потребления CPU выше порогового значения и автоматически увеличил количество реплик — в какой‑то момент их стало больше двух (вплоть до максимума, заданного в HPA).

Сразу после окончания теста, когда нагрузка спала, контроллер HPA постепенно уменьшил число подов обратно до исходных двух реплик. Это наглядно подтверждает работоспособность автомасштабирования.

## Слайд 7. Возможные улучшения проекта

Рассматриваем несколько направлений для развития системы:

- **CDN для статики**  
  Вынос фронтенда (HTML, CSS, JS) в Content Delivery Network ускорит загрузку страниц для пользователей из разных регионов и одновременно снизит нагрузку на наш Ingress и кластер.

- **Мониторинг с Prometheus + Grafana**  
  Prometheus будет собирать метрики с каждого сервиса и Redis (загрузка CPU, память, количество запросов, ошибки), а Grafana отобразит их в удобных дашбордах. Это позволит быстро обнаруживать аномалии, прогнозировать потребность в ресурсах и оперативно реагировать на инциденты.

- **CI/CD pipeline (GitLab CI)**  
  Автоматическая сборка образов при каждом пуше в репозиторий, прогон unit‑тестов и автоматический деплой (`kubectl apply`) в кластер. Такой пайплайн исключает ручные операции, снижает риск ошибок и сокращает время доставки изменений до пользователей.

## Слайд 8. Будущая архитектура (с улучшениями)

На слайде показана перспективная схема:

- Клиент обращается к **CDN** за статическим контентом. Если контент в кеше отсутствует (cache miss), CDN проксирует запрос к нашему Ingress, который уже направляет его на фронтенд внутри кластера. Динамические запросы `/api/*` по‑прежнему идут напрямую через Ingress к бэкендам.
- Добавлен компонент мониторинга: **Prometheus** с заданной периодичностью опрашивает `/metrics` у `user-service`, `ticket-service` и Redis, аккумулирует временные ряды, а **Grafana** выступает в роли визуализатора — строит графики, алерты и дашборды. Инженер одним взглядом видит состояние всей системы.
- Рядом с репозиторием **GitLab** развёрнут CI/CD pipeline: при пуше в основную ветку GitLab Runner собирает новый Docker‑образ, пушит его на Docker Hub и сразу же выполняет `kubectl apply -f k8s/` на нашем кластере. Изменения попадают в прод автоматически, безопасно и предсказуемо.
